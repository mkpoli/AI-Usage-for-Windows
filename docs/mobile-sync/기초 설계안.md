# AI Usage for Windows — Mobile Sync 기초 설계안

## 프로젝트 목표

**AI Usage for Windows**가 PC에서 Claude, Codex, Gemini, Antigravity, Cursor 사용량을 수집하고, 새 iOS/Android 앱과 위젯이 Firebase를 통해 해당 데이터를 확인합니다.

> GitHub Copilot은 Windows 앱 provider 범위에 포함합니다.

> 모바일 앱은 Provider에 직접 로그인하지 않습니다.  
> Provider 인증과 수집은 PC 앱이 담당하고, 모바일은 Firebase에 업로드된 usage snapshot만 읽습니다.

## 핵심 흐름

1. 모바일 앱에서 Google / Apple / GitHub 로그인
2. 모바일 앱이 6자리 PC 연결 코드 발급
3. AI Usage for Windows Settings에서 **Mobile Sync** 버튼 클릭
4. PC에 6자리 코드 입력
5. PC가 Firebase에 device로 등록됨
6. PC가 주기적으로 usage snapshot 업로드
7. 모바일 앱과 위젯이 해당 device의 latest snapshot 표시

---

## 인증 설계

### 모바일 로그인

사용자에게는 일반 소셜 로그인만 보입니다. Firebase Auth는 내부적으로 사용자 식별용 `uid`를 제공합니다.

| 로그인 방식 | 비고 |
|---|---|
| Google | MVP 포함 |
| Apple | 2차 범위 |
| GitHub | MVP 포함 |

### PC 연동

- PC는 별도 Provider 로그인을 하지 않습니다.
- 모바일에서 발급한 **6자리 pairing code**로 사용자 계정에 연결됩니다.
- 연결 후 PC는 device token 또는 Firebase custom auth 흐름으로 해당 device 데이터만 업로드합니다.
- PC의 upload credential은 Windows Credential Manager에 저장합니다.

## PC Sync 동작

AI Usage for Windows는 내부 사용량 상태를 Firebase로 업로드합니다.

Mobile Sync가 포함되는 첫 Windows 버전은 `0.2.0`으로 관리합니다. 모바일 앱 MVP는 `0.1.0`, snapshot payload는 `schemaVersion: 1`부터 시작합니다.

### 업로드 시점

- 앱 시작 후
- provider refresh 성공 후
- 사용자가 **Sync Now** 클릭
- 주기적 자동 동기화 (기본 5–15분)
- 종료 전 best-effort 업로드

### 업로드하는 것 ✅

- provider 이름
- plan 이름
- 사용률 / 잔여율
- reset time
- last fetched time
- provider error state
- syncProtocolVersion / schemaVersion 같은 호환성 메타데이터

### 업로드하지 않는 것 ❌

- Provider access token / refresh token
- API key
- 로컬 파일 경로
- 로그 원문
- 사용자 인증 파일 내용
- Provider cookie/session 파일

## 상태 정의

| 상태 | 설명 |
|---|---|
| `Not Linked` | 연결된 PC 없음 |
| `Fresh` | 최근 15분 이내 업데이트 |
| `Stale` | 15분 이상 업데이트 없음 |
| `Offline` | 24시간 이상 PC 업데이트 없음 |
| `Provider Error` | 특정 provider 수집 실패 |
| `Revoked` | PC 연결 해제됨 |

## 보안 원칙

- Firebase에는 **표시용 usage snapshot만 저장**
- Provider credential은 **PC에만 저장**
- PC upload credential은 **Windows Credential Manager에 저장**
- Pairing code는 **6자리, 10분 만료, 1회 사용**
- 사용자는 모바일 앱에서 PC device를 revoke 가능
- PC는 revoke된 device로 업로드 불가
- Firestore Rules는 기본적으로 자기 `uid` 아래 데이터만 접근 허용

### Firestore Rules — 기본 방향

```javascript
match /users/{userId}/{document=**} {
  allow read, write: if request.auth != null
    && request.auth.uid == userId;
}
```

### Firestore Rules — 더 안전한 방향

- Pairing 검증은 **Cloud Functions**에서 처리
- PC 업로드는 **device token 검증** 후 허용
- **snapshot schema validation** 적용

## 버전 관리 원칙

| 항목 | 초기 버전 | 변경 기준 |
|---|---:|---|
| Windows Mobile Sync | `0.2.0` | Windows 앱에 Mobile Sync UI, uploader, credential 저장이 포함되는 첫 버전 |
| Mobile 앱 MVP | `0.1.0` | 로그인, pairing, latest snapshot viewer, widget cache 포함 |
| syncProtocolVersion | `1` | pairing/upload endpoint contract가 깨지는 방식으로 바뀔 때 |
| snapshot schemaVersion | `1` | Firebase snapshot payload 구조가 깨지는 방식으로 바뀔 때 |

Provider가 추가되어도 payload shape가 같으면 `schemaVersion`은 유지합니다. 필수 필드 추가, 필드 의미 변경, `lines[]` 구조 변경은 `schemaVersion` 증가와 모바일 fallback을 함께 설계합니다.

## 설계 방향 요약

이 설계는 Provider별 모바일 직접 인증 문제를 피하면서, 기존 PC 앱이 가진 수집 능력을 그대로 활용합니다.  
모바일은 Firebase를 통해 PC가 올려놓은 snapshot을 읽는 **read-only viewer** 역할에 집중하며,  
민감한 credential은 PC 밖으로 나가지 않습니다.
